Spider RL — 3-day autonomous experiment report

Qwen3-4B + GiGPO on Yale Spider 1.0 (SQL agent task) · Mac MLX phase: 2026-05-02 → 2026-05-05 (7 runs, ~60 GPU-h) · verl-agent / A800 phase: 2026-05-07 (v1, ~1.75 GPU-h) · v2 full-coverage: 2026-05-07 → 2026-05-08 (~10 GPU-h)
TL;DR
  • Mac MLX phase — v4 (binary reward, group_size=8, no KL, n_train=300) hit +7.6pp on dev n=1034 (44.0% → 51.6%). v7 (lora rank=16 layers=8) won on test n=128 (57.8%) by unlocking extra-hard SQL.
  • Counter-intuitive findings (all phases): (1) partial-credit reward hurts, not helps. (2) Scaling train data does not help at fixed compute. (3) KL anchor only redistributes performance.
  • verl-agent / A800 phase (v1) — switching to Qwen3-4B-Instruct-2507 + bf16 + LoRA rank=32 all layers + official GiGPO step-level grouping jumped baseline from 44% → 62.9% (mostly model swap) and pushed final dev to 75.3% (+12.4pp). Test n=2147 = 75.0%, OOD gap only 0.3pp — RL learned general SQL skills, not memorized DB schemas.
  • v2 full-coverage training — same model, but deterministic shuffled-pool sampling over all 7000 train_spider questions + 220 PPO steps (vs v1's 50). Pushed dev to 78.6% and test to 78.0% (+15pp baseline, +3pp over v1). Confirms v1's "step 10 saturation" was data starvation, not algorithmic ceiling.
  • v1 saturated at step 10, v2 saturated at step 200: the same model with same LoRA reaches different ceilings depending on data coverage.
  • Sample size matters: test n=128 (mac) has ±2-3pp variance; n=64 in-training val swings 0.49 → 0.81. Only n≥1000 ranks reliably.

1. Setup

Task

Multi-turn SQL agent on Spider 1.0: given a SQLite schema and a natural-language question, the agent issues exploratory sql[…] queries (read-only execute, returns up to 10 rows) and finally commits via answer[…]. Reward is execution accuracy (EX) — predicted query's result-set matches the gold's. Order-sensitive only when ORDER BY/LIMIT is used.

Datasets

  • Train: train_spider.json (7000 questions, 30 DBs)
  • Dev: dev.json (1034 questions, 20 DBs from same set)
  • Test: test.json (2147 questions, 206 unseen DBs) — gold released by Yale 2023 after closing the leaderboard

Common config (all runs)

model:    Qwen3-4B-MLX-4bit
lora:     rank=8 layers=4 (v2-v6); rank=16 layers=8 (v7)
algorithm: GiGPO (mean_norm)
lr:       5e-5
kl_coef:  0.0 (v2-v4, v6, v7); 0.001 (v5)
ppo:      epsilon=0.2, epsilon_high=0.28, clip_c=3.0
batch:    8 prompts × group_size trajectories
group:    4 (v1, not run); 8 (v2-v7)
max steps: 6 turns / max_tokens: 384
val:      temp=0.4 on held-out train tail [6000:6128]
eval:     temp=0.0 (greedy) on dev/test

Infrastructure built during the 3 days

  • SQLAgentEnvironment with test split routing + partial_credit toggle
  • Policy.save_adapters / load_adapters — LoRA round-trip via mlx.save_safetensors; v1 lost weights because no save logic existed
  • Trainer auto-saves best/ on val improvement, plus step_*, epoch_*, final/
  • Per-hardness eval script (inline AST hardness scoring, no nltk dependency)
  • 27 unit tests covering action parsing edge cases (bracket mismatch, slop normalization, partial credit toggle)
  • Lark webhook watcher pings on val ticks / new-best / errors / exit

Result: dev n=1034 EX 51.6% (+7.6pp vs baseline 44.0%). Test n=128 EX 55.5%.

Where v4 helps most

  • Medium (50% → 62%, +11.7pp) — multi-condition WHERE, GROUP BY, simple ORDER BY/LIMIT
  • Easy (60% → 70%, +10.1pp) — single-table SELECT/COUNT, the common case
  • Decisiveness: agent commits faster — avg 2.54 → 1.97 steps per episode (-22%); answered rate 90% → 99%

Where v4 doesn't help

  • Hard (31% → 29%, -2.3pp) — nested subqueries, the agent's "fast commit" tendency hurts on queries needing more exploration
  • Extra (18% → 21%, +3.6pp small) — INTERSECT/EXCEPT/multi-table joins, capacity ceiling

4. Counter-intuitive findings

Finding 1: partial credit reward HURTS

v2 (partial=0.1) → v4 (partial=0): +2.4pp test, +9.3pp on medium.

I added a 0.1 reward for "valid SQL with wrong result" expecting it to give GiGPO a denser signal. Instead, the agent learned to settle for syntactically valid but logically wrong queries — small reward → small gradient → no further improvement. With binary 1.0/0.0, the agent is forced to find truly correct queries; result-side: avg_steps drops from 2.49 to 2.08 (much more decisive).

This contradicts the standard "dense reward helps RL" intuition. For tasks where partial-credit candidates outnumber fully-correct ones (Spider has many "looks-right" queries that miss a column), partial credit creates a false plateau.

Finding 2: more training data does NOT help (at fixed compute)

v3 (n=600) and v6 (n=400) both underperform v4 (n=300).

I expected scaling train from 300 → 400 → 600 to give 1-3pp lift via more diverse DBs. Both runs plateaued at the same ceiling as v4 or below. Likely cause: at fixed batch_size and epoch count, n=300 gives 76 PPO updates per 2 epochs — already enough to saturate what RL can extract from this base model + LoRA capacity. Doubling data without doubling updates dilutes signal-per-batch.

Implication: when scaling, scale both data and update count, or instead increase capacity (LoRA rank/layers) — see v7.

Finding 3: KL anchor REDISTRIBUTES, doesn't help

v5 (v4 + kl=0.001): same overall as v2 (53.1%), but easy 70% / extra 35% are highest of all runs.

Adding a small KL anchor was supposed to stabilize v4's volatile val curve. It did stabilize — and that cost the aggressive generalization on medium/hard. The gains on easy/extra (where the base model is already weak) come from preserving its existing strengths. Net: zero overall change.

Interpretation: KL anchor controls how far the policy can drift from the base. Drift = ability to explore new strategies. For hard reasoning gains, you need drift; for protecting niche strengths, you need anchor. The 0.001 setting biased toward "anchor" too much for this task.

Finding 4: bigger LoRA helps the hardest buckets, hurts the volume buckets

v7 (rank=16, layers=8): test n=128 +2.3pp vs v4. But dev n=1034 -1.1pp.

Per-bucket on dev: v7 picks up hard +5.2pp and extra +4.2pp, but loses medium -5.6pp. Larger LoRA (4× trainable params) lets the agent learn deeper schema-reasoning behaviors useful for nested queries, but its policy on common (medium) queries becomes less reliable — possibly because the same large capacity also overfits.

Practical implication: use v4 as default; switch to v7 if downstream task is hard-skewed.

Finding 5: small sample variance is large enough to flip rankings

Test n=128 ranking: v7 > v4. Dev n=1034 ranking: v4 > v7. The same checkpoints, the same eval procedure, only sample size differs. ±2-3pp is normal at n=128. Always confirm on n≥1000 before claiming a winner.

5. Per-run details

v2 — first attempt with partial credit (test 53.1%)

Config: partial_credit=0.1, group=8, n_train=300, epochs=2.

Val curve: 39 → 41 → 40 → 44 → 44 → 41 → 44 → 40 (best @ step 30). Stable +5pp dev signal, +3.1pp test. First reproducible RL win on Spider after the bracket-parsing env bug fix (extract_action regression that was silently dropping ~5/128 valid answers).

v2 training curves. Top-left: train c (correct rate) ma10 climbs from 0.40 → 0.55 peak then decays. Val (red) tracks 0.41-0.45.

v3 — scale up data, kept partial credit (test 52.3%, stopped early)

Config: v2 + n_train=600. Stopped after 60/150 batches once train ma c regressed to baseline and val plateaued at +2pp. Test dropped 0.8pp vs v2 — scale-up did not help with partial credit on.

v4 — kill partial credit (test 55.5%, dev 51.6% ★)

Config: v2 - partial credit (binary reward). Val curve: 42 → 40 → 39 → 45 → 43 → 44 → 38 → 39 (best @ step 30). Volatile but the best/ checkpoint locks in the peak. Hard +6.3pp on test (vs v2's +6.3pp — same), medium +9.3pp.

v4 training curves. Train c hits 0.57 peak earlier than v2 then settles. Val won_rate (red) shows clearer +5pp lift.

v5 — add small KL anchor (test 53.1%, stopped early at batch 58)

Config: v4 + kl_coef=0.001. Stopped early after pattern showed: val flatlined at 41% from steps 20-50, no breakouts. Final test 53.1% same as v2. Per-hardness shows the redistribution effect — easy 70 / extra 35 are best across all runs but medium/hard fall back.

v6 — scale up data without partial credit (test 51.6%, stopped early)

Config: v4 + n_train=400. Stopped after 80/100 batches. Best val 0.425 (step 40), well below v4's 0.451. Confirms that scale-up doesn't help even when partial credit is off — it's not a partial-credit interaction effect.

v7 — bigger LoRA (test 57.8%, dev 50.5%)

Config: v4 + lora_rank 8→16, layers 4→8 (~4× trainable params). Val curve: 42 → 39 → 35 → 45 → 47 → 42 → 45 → 45 (best @ step 40). Higher peaks AND deeper dips than v4 — more capacity = more PPO variance. Test n=128 winner at 57.8%, but dev n=1034 puts it 1.1pp below v4. Recommended only when the downstream workload is hard/extra-skewed (where v7 dominates +5pp).

7. Open questions / next experiments

  1. v4 reproducibility — single run, no seed control. Re-run with different RNG state to estimate run-to-run variance on dev.
  2. v4 + 3+ epochs — does more iteration on n=300 help? v6 showed scaling data doesn't, but iteration might.
  3. v4 vs v7 on full test n=2147 — would resolve the test-vs-dev ranking discrepancy definitively.
  4. lr sweep — only 5e-5 tested. lr=1e-4 might unlock more from binary reward; lr=2e-5 might stabilize v7.
  5. Larger group_size (16) — bigger groups give cleaner GiGPO advantages. Cost: 2× wall time.
  6. v4 + warmup schedule — early-batch volatility (steps 0-20) might smooth out with cosine warmup, possibly raising the best/ peak.

10. Update: verl-agent / A800 phase (2026-05-07)

v1 — verl-agent + A800 80GB
  • Dev n=1034 best: 75.3% (step_30) — +23.7pp over mac v4 on the same metric
  • Test n=2147 best: 75.0% (step_30) — OOD gap only 0.3pp = generalizes to 206 unseen DBs
  • Cost: 1.75 GPU-hours training (vs 60 GPU-h on mac); 1 GPU-hour for full eval
  • Saturation: step 10 already at 74.8% — RL learning is fast and shallow at this LoRA scale

Why we moved off mac MLX

  • Hardware: Mac MLX 4-bit Qwen3-4B is ~9 min/batch with group=8; A800 80GB hits the same step in ~100 sec → ~5× faster per step
  • Quantization: MLX 4-bit base imposed a quality ceiling. bf16 base is materially better at SQL reasoning out-of-the-box.
  • LoRA capacity: 8 GB GPU couldn't comfortably fit larger LoRA + KV cache + FSDP staging. 80 GB lets us run rank=32 across all 36 transformer layers (41M trainable params, ~63× the mac v4 setup).
  • Algorithm: verl-agent is the official GiGPO implementation (NeurIPS 2025) with step-level grouping; mac was a custom GiGPO that only had episode-level grouping.

v1 config

Decomposition of the +23.7pp jump from mac v4 best to verl-agent v1 best:

  • ~+18.9pp from base swap (Qwen3-4B-MLX-4bit → Qwen3-4B-Instruct-2507 bf16): the base-only delta on dev is 44.0% → 62.9%. The bigger newer base (released 2025-07) plus dropping 4-bit quantization is the dominant factor.
  • ~+4.8pp from RL recipe upgrade: gain widened from +7.6pp to +12.4pp under the same algorithmic family. This is attributable to:
    • LoRA capacity 0.65M → 41M params (×63)
    • verl-agent's official GiGPO with step-level grouping (mac had episode-level only)
    • Lower LR (3e-6 vs 5e-5) — verl-agent's tested default for RL on instruct models

What didn't change (mac findings still hold)

  • Binary reward beats partial credit — kept partial_credit=0 in v1. Same logic: small partial reward → small gradient → settles for "looks-right" SQL.
  • KL anchor disabled — verl-agent's webshop_lora.sh default uses use_kl_loss=False; we kept it. Did not test KL on/off ablation in v1.
  • Group_size=8 — same as mac v4.
  • n=64 val noise is real — v1 step_50 reported 0.852 val (n=64) but actual dev n=1034 = 0.750. Mac findings about small-sample variance reproduced.

v1 open questions / next experiments

  • Break the 75% plateau: train longer (200 steps), increase batch size (8 → 16 questions), or apply difficulty filtering (only 20–80% solve-rate questions) to escape the early-saturation regime.
  • Pass^k consistency — single-pass 75% but how many of those are reproducible at temp=1.0? Mac v4 had 60.9% pass@1 / 35.9% pass^5 (huge consistency gap); v1 hasn't been measured.
  • Capacity ablation: rerun v1 with mac's rank=8 layers=4 LoRA to isolate "verl-agent algorithm" vs "LoRA scale" contribution. Hypothesis: most of the +4.8pp algorithm-side delta comes from the larger LoRA, not GiGPO step-level grouping.
  • KL ablation: try kl_coef=0.001 and kl_coef=0 — Composer 2 (Cursor 2025) uses KL with k1 estimator on 1T MoE, but mac v5 found KL hurts on 4B. Verify on verl-agent setup.
  • SFT cold start — distill ~1k Qwen3-72B chain-of-thought + SQL trajectories onto Qwen3-4B before RL. Could help break past 75% by giving the model a stronger reasoning prior than the base instruct tuning provides.

v2 in-training val noise

Critical lesson: n=64 fixed-prefix in-training val is unreliable for picking best checkpoint. Despite using fixed first 64 dev questions for stability, in-training val showed step_90 = 0.895 as peak. Real dev n=1034 showed step_200 was actually best (0.786). The n=64 sample drifted in 0.82–0.90 stochastically — useful for trend-watching, useless for ranking checkpoints.

Implication: always run full-dev eval post-training. Don't pick the "best" checkpoint based on in-training val score alone.

v2 plan that didn't pan out

The hypothesis: scaling train data 6% → ~50% coverage would push past v1's plateau by 5–10pp. Outcome was +3pp — real but smaller than expected. Two corrections to the mental model:

  • Saturation isn't purely about data size. Once model finds the common SQL patterns, more diverse questions yields diminishing returns. Full Spider coverage hits the same wall as random 6% coverage, just slower.
  • Per-step gradient noise dominates late training. n=64 val swings 0.82–0.90 (±5pp) randomly, suggesting trajectory variance is high. Bigger group_size or KL anchor might tighten this.

  • P0 — Run full 438 steps (not 220 like v2): with KL anchor in place to prevent overfit. Cost: ~$30 GPU. Predicted: +0–1pp (might just plateau).
  • P1 — LoRA rank 32 → 64: 2× capacity, useful if hard/extra bucket is the bottleneck. Cost: ~$30 GPU (1.5× slower). Predicted: +1–3pp on hard subset.
  • P1 — Difficulty filter: pre-compute solve rates on base model, train only 20–80% mixed-rate questions. Cost: 4 hr filter build + retrain ~$50. Predicted: +2–3pp via cleaner gradient signal.
  • P1 — Switch to Qwen3-7B base: bigger backbone. Cost: 80GB GPU is tight; ~$60 GPU. Predicted: +5–10pp (would put us at GPT-4 level).
